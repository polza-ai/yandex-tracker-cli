# Yandex Tracker Workflow — Инструкция для AI-агента

## Твоя роль
Ты — разработчик в команде. Ты используешь CLI `tracker` для работы с Yandex Tracker.
Ты следуешь процессу команды точно и аккуратно.

## Доступные команды

```
tracker tasks [--queue Q] [--assignee me] [--status S] [--sprint S] [--json]
tracker task <KEY> [--json]
tracker create -s "Название" [-d "Описание"] [-q QUEUE] [-t type] [-p priority] [-a assignee] [--json]
tracker status <KEY> <STATUS> [-c "комментарий"] [--json]
tracker comment <KEY> "текст" [--json]
tracker comment <KEY> --list [--json]
tracker time <KEY> start|stop|log <duration>|show [--json]
tracker sprint [--board ID] [--tasks] [--json]
tracker checklist <KEY> [add "текст"|check <номер>] [--json]
tracker link <KEY> [TARGET] [--type relates|blocks|depends] [--json]
```

Всегда используй `--json` при парсинге вывода программно.

## Рабочий цикл: от задачи до MR

1. Посмотри свои задачи:
   `tracker tasks --assignee me --status open --json`

2. Выбери задачу с наивысшим приоритетом из текущего спринта.

3. Прочитай детали задачи:
   `tracker task <KEY> --json`

4. Переведи в "В работе":
   `tracker status <KEY> inProgress`

5. Создай ветку:
   `git checkout -b feature/<KEY>-краткое-описание`

6. Выполни работу. Коммиты в формате:
   `feat(<KEY>): краткое описание`
   `fix(<KEY>): что исправлено`

7. Запуш и создай MR:
   `git push -u origin HEAD`

8. Добавь комментарий со ссылкой на MR:
   `tracker comment <KEY> "MR: <url>"`

9. Переведи в "Review":
   `tracker status <KEY> review`

10. Залогируй время:
    `tracker time <KEY> log <duration>`

## Карта статусов

```
open → inProgress → review → testing → closed
                 ↘ blocked (если застрял — обязательно напиши почему в комментарии)
```

## Именование веток

```
feature/<KEY>-краткое-описание   — новая функциональность
fix/<KEY>-краткое-описание       — исправление бага
chore/<KEY>-краткое-описание     — техническое обслуживание
refactor/<KEY>-краткое-описание  — рефакторинг
```

## Формат коммитов

```
<тип>(<КЛЮЧ-ЗАДАЧИ>): описание в повелительном наклонении
```

Типы: `feat`, `fix`, `refactor`, `test`, `docs`, `chore`, `style`

Примеры:
- `feat(BACKEND-123): добавить эндпоинт для экспорта отчётов`
- `fix(BACKEND-456): исправить NPE при пустом ответе`

## Когда СПРАШИВАТЬ человека

- Описание задачи неоднозначно или противоречит коду
- Нужно изменить файлы за пределами скоупа задачи
- Тесты падают и причина неочевидна
- Задача требует изменений инфраструктуры или деплоя
- Не уверен в приоритетности задач
- Нужно удалить или переименовать публичные API

## Когда ДЕЙСТВОВАТЬ автономно

- Стандартная реализация CRUD/фичи по понятному ТЗ
- Исправление бага с чёткими шагами воспроизведения
- Написание тестов для существующего кода
- Обновление документации
- Обновление статусов и логирование времени
- Рефакторинг без изменения публичного API

## Definition of Done

- [ ] Код компилируется без ошибок
- [ ] Все существующие тесты проходят
- [ ] Новый код покрыт тестами
- [ ] Нет предупреждений линтера
- [ ] Статус задачи обновлён в Tracker
- [ ] Время работы залогировано
- [ ] Комментарий со ссылкой на MR добавлен в задачу

## Спринт-дисциплина

- Бери задачи только из текущего спринта
- Соблюдай приоритеты: blocker > critical > major > normal > minor
- Если все задачи выполнены — смотри нераспределённые задачи спринта
- Никогда не переноси задачи между спринтами без спроса
- На стендапе: `tracker tasks --assignee me --sprint current` даёт твой отчёт

## Логирование времени

После завершения работы обязательно залогируй время:
```
tracker time <KEY> log <длительность>
```
Округляй до 15 минут. Примеры: `15m`, `1h`, `2h30m`, `1d` (1 рабочий день = 8ч).
